What to expect from this Essay

I will clear up common misconceptions, explain what’s important for a first programming language, introduce the most interesting languages and point to the best (free) learning resources and books.

The easy and the hard Part

“Learning a programming language” is what applies foremost to programmers who already know how to program. For them, picking up a new language is possible within a couple of weeks (and for you too, once you’ve gained some experience), and it does’t matter really that much which one – because the patterns how to accomplish most things are roughly overlapping.

But if you start out, you don’t need to just learn a programming language. First, you want to learn how to program. It means learning how to analyze a problem you want to tackle, finding a solution and express it in code, which is often not intuitive. Plus, you still have to learn a programming language with syntax and semantics, too. So that is two things at once.

Your Reason to learn coding

Do you want to prepair for a career as a software developer? There are different areas where software is developed – the most obvious within the web, is obviously the development of the web itself. But there are other areas like fintech, aircraft systems, robotics, telecom, water supply or internet of things. If you know exactly what you intent to do, you could start with a language of your specific field and go form there. Or maybe better not?

“It’s crucial that you understand
the principles behind programming
that transcend the specifics
of any specific language.”

You should go with a language that’s interesting to you – for whatever reason – and is particularly suitable for learning, according to the criteria I lay out below. Some are bad for learning, for various reasons. I’m also going to describe the most interesting languages and their idiosyncrasies.

You may want to learn programming, even if you have no plans to become a part of the software workforce, like me. I’ve worked quite a long time as a designer and in online marketing industry, but have barely written any code at all, since there were always developers working alongside me and tasked with the wizardry.

But I have always been interested in computer stuff, so I’ve dabbled a lot with Linux, server administration and tweaking my configs as a means of procrastination. Real programming was the magical border I haven’t crossed – until recently.

For my part, I want a tool in which I can think, and I’d like to be able to prototype an idea if I have one. Further, I think programming can help us tremendously to develop our minds. I must say that hated mathematics in school and have been really bad at it, but I’m fascinated and eventually I want to know what is this all about. So I had to start somewhere.

and wield some magic!”

What makes a programming language beginner-friendly?

Available learning Material

For which programming language is the right learning material available? This point is huge! I’d go so far and choose a first language solely on this single criterion.

Most programming books or courses are not made for absolute beginners, but for programmers who want to learn another language. Remember, you want to learn programming, not just a language (You’re going to learn the language on the go, and it will not be your only one).

It’s not that “regular” programming books are “too advanced” to grasp; it’s just that they teach you the wrong things, instead of the things you’ll actually need to learn.

I think it would be wise to prefer material from university profs and/or other reputable personalities to make sure you adopt a well rounded foundation and good style, instead of jumping between random tutorials. The more popular the language, the more crap you’ll run into.

That’s why I recommend a textbook or e-textbook with exercises as your main forge of knowledge, because books are still the most accessible media – you can read a chapter even if you’re queueing up somewhere.

Prepare to dabble with your textbook for quite some time. So you really want an enjoyable one! The most recommended book may not neccesarily be the right one for you; individual taste matters a lot. I’ve listed the best available programming books below with some alternatives, for each language.

What kind of animal are you?

Programming languages are created by different minds, within different eras, significantly influenced by fashion. One language may be a natural fit for a certain personality type, and a less fortunate choice for another. That even applies to programming itself – it’s easier for some and harder for others (doesn’t mean those are more stupid).

It seems there are two different groups of programmers. Those that see programming languages primarily as a means of production (category 1) and those who see it as instrument of human reason (categories 2 and 3):

  1. Pragmatic languages: conservative, efficient, boring.
    “A programming language is a tool to get the job done”
    (C, Java, Go)
  2. Hackish languages: multi-paradigm, anything goes, unopinionated.
    “I pursue absolute freedom and like to explore the problem domain interactively”
    (Lisp, Scheme, Javascript)
  3. Constraining languages: single-paradigm, protecting you from yourself, opinionated.
    “I prefer one correct way according a fixed set of rules”
    (Elm, Haskell, Python)

Which programming language will be in high demand

… in late autumn 2026 so I can get filthy rich coding in it? Well, nobody knows and it doesn’t matter. Why? Because it’s about learning to code – which is a timeless skill – not about a particular language (those come and go).

Programming is like being able to ride a bicycle. Once you’ve mastered the underlying principle, you won’t die instantly when you fall over, and it will not make much of a difference if you hop on Grandma’s grinder or shred your neighbour’s kid’s tricycle, and it will not matter much if the current trend says in 10 years everyone is riding nothing but mountain bikes with pink tyres.

But! There’s one thing that’s getting more and more important. It’s called concurrency. Most languages in use today have been developed years ago when Moore’s Law was still valid. Back then, everyone expected single processors getting faster and faster, so it was perfectly fine to have languages to make programs running on a single processor, and no one cared to build in concurrency features.

But that is not any more. To keep up with the demands of modern applications, we have no other chance than to put more processors into a computer, or to run the application distributed on multiple computers. To leverage the power of multi-core processors, we need to be able to program our software conveniently in a way to utilize all available cores.

That leads us to learn a programming language that is designed with concurrency in mind and simplifies building concurrent software.

Community and Popularity

Popular languages have some benefits – for instance bigger ecosystems, which leads to many things built around the language to make the life of (experienced) programmers easier.

Libraries are part of the ecosystem: those are pieces of code written by other programmers that can be re-used, so a developer has less work and doesn’t need to code everything from scratch.

There are programming languages that have strong feelings attached and attract a fanatic cult-like following, more than others. Those are Python, Haskell, and Lisp.

Obviously, popular languages have also bigger communities. You don’t need a huge community, but it should be a welcoming one. Smaller communities have their benefits, too.

So, 700,000 people fullmoon rave or beach party with friends? Meh, forget both and practice some social distancing – it’s quite popular in high demand right now!

Expressive Power and Abstraction

Expressivity means how simple it is to say certain things. There are a couple of factors who determine how expressive a programming language can be:

Low-level or High-level?

Low-level does not mean “worse” or “inferior” than high-level – it just refers to the fact that computers are built in many layers on top of each other, and the levels close to the physical machine are called the “low level”. You could also call it “foundation”, if you prefer a less judgementally sounding term xP

High-level languages tend to be more expressive than low-level languages, because with each layer up, the grade of abstraction increases and more powerful ways to express complex things come into play.

How simple is it to build Abstractions?

What is abstraction? It means hiding the foundational workings of the level(s) below (eg. the physical machine) so you don’t have to deal with them directly and can focus on your problem/idea instead.

An analogy: A console used to control the lightshow of a concert is an abstraction over fumbling with the electricity cables and tieing them together to make light bulbs glow in a certain order. Now think about doing the light show only by messing with the wires. This is dangerous and way harder (nearly impossible) than using the console. And so it is with programming, too.

Is it a wordy or concise Language?

Do you have to write a lot of boilerplate or repeat yourself? In most cases you want an expressive high-level language; not overly verbose but also not too concise, so there’s just the right amount of redundancy left to keep it comprehensible.

Side-read: programming languages ranked by their expressiveness

Syntax and Consistency

As a novice coder, you probably want to start out quickly, and you’ll need enough brain capacity left to maintain focus on the important things, without the language getting in your way. So you want a simple syntax that’s easy to pick up and remember.

A thoroughly designed programming language suits well for learning, because you will encounter less surprises and pitfalls. Consistency helps you to derive further things from things you already have learned before.

Maturity

While it may be tempting to start with the current hyped language everyone is raving about, a mature programming language has the benefit of an organically grown user base, flaws ironed out, stuff being written about it, better tooling, etc.

A mature language doesn’t change every couple of months, so you can be sure to learn the “right” things – you don’t have to worry if you’re learning old stuff without knowing.

Which language has been around for quite a while, but implements modern, interesting and useful concepts that may be relevant in the future?

Tooling and Usability

Your language of choice should come with tooling that’s easy to use. It includes things like editor support (syntax highlighting, “jump-to-definition”, looking up documentation), helpful (or at least comprehensible) error messages, and package management for libraries.

A good selection of libraries is desirable for industrial use, but doesn’t matter sooo much in the beginning. Some languages can even use libraries from other programming languages / ecosystems, and go even further than that.

Consider to pick a language that comes with a REPL, sometimes called console or toplevel. That’s basically like the command line running in a terminal emulator, but instead of shell commands you can run single pieces of your code. Well, nowadays many languages come with a REPL. But some go even further than that and enable true interactive development.

Can you get started without fumbling for days?

Choose your Poison

Here is a carefully picked selection of interesting programming languages. Some of them are better teaching languages, while others are worse.

ANSI C

Welcome to the low-level. Programs written in C tend to be much more performant than programs written in other languages. In other words, it is the fastest (practical) programming language. Performance-critical algorithms and other parts of applications are usually written in C; e. g. the Linux kernel, drivers, high-performance libraries, etc.

There seems to be a tradeoff with programming languages:

  • either high run-time performance (fast) plus low programmer productivity → longer development time
  • or lower run-time performance (slow) but higher programmer productivity → shorter development time

Development time (human) is more expensive than run-time time (computer), so mostly we agree on a slower programming language to minimize the human effort. However, since performance is a matter of implementation, not the programming language per se, this is not entirely true. There are programming languages coming close to C performance, one example is Julia

You may hear here and there that C is the foundation for everything and everyone must learn programming in C before even thinking about anything else – but no, don’t waste your time: Learn C if you have a genuine interest in systems programming or another reason to work very close to the machine.

If that is the case, and you wanna become a hardware hacker, you may also look into the Arduino electronics platform, coming with the Arduino programming language (which is basically a subset of C). Don’t confuse C with C++ (an extension to C, but huge and messy) or C# (a Microsoft thing). Look for “ANSI C” – the standard.

  • C programming book for beginners:
    • The C Programming Language, 2nd Edition (paper/Kindle)
    • Beginning C for Arduino, Second Edition (paper/Kindle)

Lua

On the surface, Lua is quite similar to JavaScript – just without the ugly parts (more about that later). Lua is elegant, performant, carefully designed and consistent. Therefore it is super easy to learn for Muggles Non-programmers, and probably the “easiest” programming language listed here.

The language is mainly used in the game industry (e. g. World of Warcraft, Roblox). Lua goes very well with C and it can be embedded into C code, so that high-level program logic – like character behavior – can be scripted in Lua, while the other parts of the software in need for raw metal speed can be written in C or C++.

Lua can be used everywhere and it’s quite simple to get started and build something with it: You might want to make your own 2D game with the Löve game engine, you can use it to code native games and other apps for Android. There’s a really nice IDE (“Integrated Development Environment”) for Lua programming, called “ZeroBrane Studio”; think of it as an extended editor especially for Lua.

As far as I know, there are not many textbooks available, but you can use the comprehensive book written by Lua’s chief architect Roberto Ierusalimschy. And there’s a wiki and many many user tutorials. Lua is primarily developed at the University of Rio de Janeiro (PUC-Rio). Once you’ve commited to learn Lua, I recommend to buy the 4th edition to support the development.

  • Getting started with Lua, ZeroBrane Studio and the Löve game engine
  • Lua programming books for beginners:
    • “Programming in Lua 1st Edition” (free online)
    • “Programming in Lua 4th Edition” (e-book) | Paper book

Clojure

As a total noob, I’ve been looking for a language with a simple syntax, because who wants to run into syntax errors all the time? Clojure is a member of the Lisp family; those have the cleanest syntax of all programming languages. Further, it seems to be the most expressive general purpose language, and it’s designed especially with concurrency in mind.

The language is built on top of the Java platform. Software built on Java can run on almost any major operating system – Windows, Linux, Mac, etc. Clojure can be used for different things, even for artsy stuff. Just as an example: you can do live coding and make algorithmic music, and there are things like algo-raves (yup, that exists …).

It’s also a practical language to build serious business software with, because it can use Java’s library ecosystem. And there’s ClojureScript, a dialect of Clojure that compiles to JavaScript (so you you won’t have to write web apps in JavaScript).

But there’s that Java underbelly who sticks out everywhere, coming with it’s own terminology. And that’s the drawback that makes it quite unsuitable as a first language: You’ll have to jump into the Java abyss eventually and learn that stuff to understand the Clojure docs and error messages, distracting you from your mission to learn programming and making it unneccessarily complicated.

If you want to try it anyway, I recommend the book “Clojure for the Brave and True”; it had given me a few enlightening moments how all the parts interlock: functions taking functions and spitting out functions, like interlocking parts of a imaginary robot factory building little machines building little machines … But I think that’s more a property of functional-first languages in general. The book convinced me also to take a look at Emacs.

  • Getting started with Clojure
  • Clojure programming book for beginners:
    • “Clojure for the Brave and True” (paper/Kindle)

Scheme and Racket

Pleasantly surprised by Clojure, I’ve looked up other languages from the Lisp family. Scheme is commonly recommended to beginners, and those who are serious about computer science. It’s a bit opinionated to be unopinionated, and Scheme minimalist (no distraction) and often considered as the most elegant and consistent programming language in existence. Therefore it is excellent for learning.

Racket is a descendant of Scheme (it was called PLT Scheme before). It is “Scheme with batteries included”, a platform for research and to create other programming languages. Scheme can run on Racket, so even if you just want to learn the Scheme language, you can do it comfortably in Racket. That may sound a bit confusing, but it actually isn’t. You’ll see.

Further, Racket emphasizes “language oriented programming”. It means you can design your own small (domain specific) programming languages for your application and then write your application in that language. But you don’t have to – you can just use “normal” Racket like everyone else. So don’t bother with that for now.

The most recommended book on (functional) programming is the timeless classic “Structure and Interpretation of Programming Languages”, abbreviatet “SICP”. It was written as an introduction for computer science students at MIT, using Scheme. But for me it was too much as a very first book on programming. But check it out anyway, it’s really philosophical.

I’ve also started to read “How to Design Programs”, which is supposedly easier and often recommended to read before SICP, but found it earnest and dry. The other books are quite different. Personally I recommend “Simply Scheme”, “Realm of Racket” (the most entertaining) and “The Little Schemer”.

You can get started very easily with Scheme and Racket – one single install and go, no matter if you are on Windows, Linux or Mac. You don’t even have to be familiar with the command line, because it comes with DrRacket, a friendly and easy to use programming environment (IDE).

    • “Realm of Racket: Learn to Program, One Game at a Time!” (paper/Kindle)

Common Lisp

The first reason for me to use a programming language wasn’t coding, actually I wanted a better calculator. A REPL makes a way better calculator than the silly concept of an interface resembling a physical pocket calculator with big buttons and a single line display.

Well, here Lisp’s functional notation felt really weird for doing arithmetik: Instead of 10 + 3 + 4 + 6 + 7 you write (+ 10 3 4 6 7). There are no operatiors, it’s basically all function application. All Lisp languages have this uniform and simple syntax. It makes a couple of things easier, for instance to write programs who write programs who … well, that’s for later. The syntax is easy to remember so it doesn’t get in your way, and you can concentrate on the things that really matter.

I think the most compelling books and essays have been written on Lisp, and it has the best learning material I could find. Well, Lisp has been around for a while.

What really fascinates me, is that Lisp is no “black box”, where you write some keywords and then something underneath does some things and you don’t know what and why. Lisp is different: It is built on a few simple pieces, and everything else is built from those pieces – it is transparent and easy to paint a mental picture how it actually works.

The tooling is excellent and designed for interactive development. There’s a really nice App for iOS and Android called “CL REPL”. You can use it to tinker around when ever you feel like “Damn, I wish I could write some Lisp right now!” You can. It’s in fact a complete Common Lisp IDE with SLIME, the “Superior Lisp Interaction Mode for Emacs”.

But usually you want it on your laptop. I recommend Emacs with SLIME, and it’s easy to install: use Portacle, which has everything bundled together with batteries included. If you don’t know Emacs yet, do:

  1. Menu → “Options” → “Use CUA keys”
  2. Menu → “Options” → “Save options”

– then copy & paste works like everywhere else.

  • Getting started with Common Lisp and the Protacle IDE
  • Lisp programming books for beginners:
    • “Common Lisp: A Gentle Introduction to Symbolic Computation” (paper/Kindle) | Free PDF
    • “Land of Lisp” (paper/Kindle)

Erlang and Elixir

Erlang the platform is an integral building block of today’s telecommunication networks. Your mobile provider’s infrastructure runs likely on Erlang, and so does WhatsApp. Erlang is not just a programming language, but a complex framework to write forever-running network applications; you can even think of it as an operating system. The whole point behind Erlang is to build long-running, fault-tolerant systems. So it’s all about error handling and staying online.

To achive that, Erlang leverages concurrency through an interesting concept, the actor model. That makes Elang platform a natural fit for mission-critical infrastructure, decentralized computing, peer-to-peer networking, messaging apps and servers with hundreds of thousands of simultaneous active users, internet of things … Ok ok, I see you got it.

Erlang the language and Elixir are just two languages to write applications for the Erlang platform (there are others as well, eg. LFE). Erlang (ERicsson LANGuage) is the original language. It is pragmatic, compact and easy to pick up. Erlang the language has better/mature tooling, too.

Elixir on the other hand is an alternative syntax plus some additional concepts like macros (metaprogramming). What I dislike about Elixir: Many constructs have 3-4 “shorthands” to write the same thing. How annoying! It seems the language designers couldn’t decide what to keep and what to throw out. Elixir looks a bit wordy and incoherent, forced into a Ruby appearance (that’s another programming language) to make it appealing for Ruby web developers. Here’s a comparison between the two languages.

Does it make sense to learn programming with Erlang or Elixir? Well, there is not really learning material available that makes use of those languages. You’ll find only introductions to the languages themselves (the ones I’ve listed below), so you’d miss the point.

And there is also the Erlang technology. Like with other hosted languages (Clojure on Java, Elm on JavaScript), the additional stuff coming with it may be a bit too distracting. So if you want to learn Erlang or Elixir, you should have a good reason or two to mess with the Erlang system. For a better idea what the Erlang platform is, check this out.

Erlang or Elixir or both? I’d say start with Erlang and learn the platform on the way, because the Erlang platform is explained in terms of Erlang the language, not in terms of Elixir. Elixir adds additional stuff on top of that, so you may approach it afterwards, if you see need.

  • Erlang programming books for beginners:
    • “Introducing Erlang: Getting Started in Functional Programming” (paper/Kindle)
  • Elixir programming books for beginners (same same but different):
    • “Introducing Elixir: Getting Started in Functional Programming” (paper/Kindle)

JavaScript

With JavaScript, you can build all kind of things in the browser and show it to your internet friends, without having them to download or install anything. JavaScript is the only language browsers support directly, still – and that’s why everybody and their dog’s fleas are coding JavaScript.

The language has good tooling and is easy to start with, and you may have heard “all you need is a web browser and a text editor”. Well, um, no. Unfortunately, that’s not really the truth. Expect a lot exceptions, special cases and to work around the poor language design. I hate incoherence, and JavaScript is the epitome of exactly that.

You’ll need to learn to avoid and remember all the pitfalls, and using a linter (that’s a tool that checks your code and tells you where the problematic or deprecated parts are) is mandatory to prevent you from using probably harmful things and deprecated stuff.

Overall, it’s a verbose frankenstein language that’s a pain to read. Of course there are the usual suspects who disagree and even “love” JavaScript. These are the people who either don’t know any better, or who suffer from a variant of the Stockholm Syndrome. Honestly, that doesn’t make it a good choice for a first time programmer, but despite the fact, many start to learn programming with JavaScript, unfortunally.

JavaScript is was an imperative language at the very heart, but you can write object-oriented and functional style too. JavaScript assimilates new features with each version and therefore it’s getting bigger and bigger. Resistance is futile. Or is it?

If you just wanna build web apps or browser games, you do have other choices – check out ClojureScript (scroll up) or Elm down below – eventually you need to know JavaScript, but maybe not as your first language.

Elm

Elm is a language that compiles to JavaScript: You write your website’s code in Elm, and then the Elm compiler writes the JavaScript for you, so that your web browser can run it.

If that sounds like it makes no sense, think about it this way: All programming languages compile down to another language on a lower level anyway, and those again and again, until your code’s representation is only 11010101. Elm is just another layer on top.

Why not writing JavaScript directly? Elm comes with a few interesting features: first and foremost a static type system (like Haskell), which helps you to feel (more) sure about the correctness of your code (think: less undiscovered bugs). And you don’t have to deal with JavaScripts screwed type coercion. Secondly, Elm supports the functional programming paradigm: That makes it’s easier to reason about what code actually does. And it comes with immmutable values (Ever writhed in disgust when you saw x = x + 1? Then this is for you).

The Elm syntax is simple and the Elm compiler famous for its helpful error messages – like a built-in teacher who pushes you gently into the right direction on your path to statically typed functional programming. And you will learn one way how to structure your application: “The Elm Architecture”.

That’s why I consider Elm well suited for learning, despite the fact that it’s still a very young and domain-specifig language that runs exclusively in the browser. There is not much literature out there, but Elm opens the door to other statically typed functional languages, like OCaml or Haskell.

Elm comes with an easy installation and coherent tooling. It is designed as a cathedral and quite opinionated: You’ll need to do most things like intended by the creators, similar to Python. But that has benefits too, and it may be one of the reasons why everything within the Elm ecosystem fits so well together.

  • Get startet with Elm
  • Elm programming book for beginners:
    • “Beginning Elm” (free online)

Haskell

Many of Elm’s ideas come from Haskell. So here we are. If I like Elm but it’s a browser-only language, why not learn “the real thing”?

Haskell has a simple syntax, but it’s uncommon for programmers that are ingrained with the verbose syntax of C/Java/PHP/JavaScript. However, Haskell’s syntax relies a lot on cryptic infix operators (also custom ones) and with it come things like the precendece of those operators (from 0-9). The same things can be achived by many ways in Haskell, back-and-forth.

Contrary to popular belief, basic Haskell is easy to learn; especially for a person who has never programmed before. It may be less easy for experienced programmers because of all the preconceptions they have about programming. But it’s not a compact language and Haskellers tend to be clever, so they like to use the advanced features the language and its extensions have to offer. It will get complicated very soon when you’ll want to work with libraries and encounter code from others.

Unfortunately, it’s not so straightforward to get started with Haskell. Downloading the compiler and starting ghci (the REPL) to evaluate some expressions is no big deal of course, and to compiling some code neither. But if you go a bit further and want some more convenience and install libraries, there are a couple of different approaches which tooling set to use, and they overlap in functionality, and … I think that’s really confusing.

Not all hope is lost: If you are just interested in functional programming, you’re better off to start with Ocaml (see below) or even Elm, both are better languages to learn. You can still move on to Haskell later, if you feel like. If you wanna give it a shot nevertheless, download Stack and then install everything else via Stack.

  • Getting started with Haskell (Stack)
  • Haskell programming books for beginners:
    • “Haskell - the Craft of Functional Programming” (paper/Kindle)
    • “Haskell Programming from first Priniples” (sample)

OCaml

The “(Objective) Categorical Abstract Machine Language” is used in places where performance and correctness are of utter importance, like financial technology. Two companies heavily invested in OCaml are Jane Street and Facebook.

OCaml is similar to Elm and Haskell: statically typed and emphasizing functional programming. It has less ideological ballast and is more practical than Haskell, but (still) less popular. Put another way, you could also say Haskell is overly hyped. As I see it, the language seems to be due for a rise in popularity. OCaml got a lot of support within the last time. The tooling is already way better than than Haskell’s, and there are many improvements on the way.

The user experience is really nice on Linux, and OCaml works well on Windows with WSL (Windows Subsystem for Linux), too. You may run into an issue during the installing process in WSL because of WSL’s limitations; anyway, there’s a workaround for that.

I think syntax matters – for beginners especially in terms of not getting in your way. It looks clean and doesn’t have those {curly braces} spreading all over the place, which is nice. OCaml is a bit more verbose than Haskell, but still very readable. OCaml further opens the door to F# (F Sharp), which is essentially an OCaml clone by Microsoft that runs on the .NET platform.

  • Getting started with OCaml
  • OCaml programming book for beginners:
    • OCaml from the Very Beginning (paper/Kindle) | DRM-free PDF
    • Think OCaml: How to Think Like a (Functional) Programmer (free PDF – work in progress)

Python

I’ve considered learning Python a couple of years ago, but after some research I found other languages far more elegant and interesting. It is one of the most used languages right now, amongst JavaScript and C. Popularity is probably Python’s biggest benefit. So if there’s something programmable, like say, some cheap consumer drone, then it’s likely programmable in Python.

There are Python libraries for everything, which adds to popularity. On the technical site, many of those libraries are actually written in other languages (e.g. C or C++) for performance reasons, because Python itself is by far the slowest of the languages listed here.

Python has got helpful error messages and great marketing, so everyone is learning Python right now (as of 2021). The syntax looks clean and readable, almost like broken English. It’s said to be an “easy” language, but I don’t think so. Python paints you in the object-oriented corner with all the complexity that comes with it, while it makes functional programming cumbersome (which is arguably the simpler programming paradigm, from my beginner’s perspective).

Python is opinionated: The creators encourage that certain things should (only) be done in “the right way” – which they call ’pythonic’. The opposite is ’unpythonic’. “To describe something as ‘clever’ is not considered a compliment in the Python culture”. Well, you may be more pragmatic and don’t care, but I think a beginner should not lock his mind into ’the pythonic way’ but start with a language that opens the mind. If you want a scripting language which is compact, even easier to learn and really fast, you may have some fun with Lua.

  • Getting started with Python
  • Python programming book for beginners:
    • Automate the Boring Stuff with Python (paper/Kindle)

Julia

Julia is developed as a high performance vehicle for technical and scientific computing, like data science, simulations, as an alternative to Fortran, R, Python and other languages in the field.

It’s still a young language, but pushed by a couple of organizations. So there’s more of an ecosystem than you would expect for a language in that early state. Most scientists already how to program their stuff, so don’t expect much introduction material for total beginners right now.

I’ve considered to learn Julia because there are some interesting aspects, but to me I see no purpose to learn it right now and therefore didn’t look that much into it. With Julia you can do some things that you can do with Lisp, but contrary to Lisp, Julia neither uses prefix-notation syntax nor nested parantheses.

Julia looks friendly and easy to start with, and you should definitely check it out if you’re a student have to do with statistics, numerical computing or want to plot some stuff (Racket shines here, too). I think Julia could be a very promising general-purpose language.

  • Getting started with Julia
  • Julia programming book for beginners:
    • “Think Julia: How to Think Like a Computer Scientist” (paper/Kindle) | Free online

Conclusion

Choose well, but don’t obsess too much about it: I’ve already done that part for you! You are going to learn a couple more programming languages after your first one, so it’s not a lifetime decision, even if you feel like that now.

Some courses and books teach programming using stripped-down teaching languages, or even pseudocode. I think it’s not a good idea, because you cannot use the teaching language to build anything useful along the way; you learn something you have no use for later on; and the teaching languages are often not subsets of a real programming language, which may lead to confusion thanks to a slightly off terminology.

Mkay, but which one now?

After all, I think Racket ticks most boxes if you strive for a deeper understanding of programming, computing and the bigger picture. Language design and usability are awesome, the syntax crystal clear, it has a comprehensive documentation, and Racket can teach you all the mind expanding concepts. No wonder, it’s actively used for computer science and programming language research. It’s also a bit academic. Oh, I said the A-word xP.

And there is Lua. Here, the same nice things are to say about syntax, language design and documentation. It may not teach you all those fancy concepts like Racket does, but Lua is minimalist, so you’ll be able to fit the whole language in your head, which is quite impossible with the other languages. Lua is also very practical and you can use it nearly everywhere.

And – as much as I hate to say it – but JavaScript is also an option, but for pragmatic reasons. After all, it all depends on your individual priorities. JavaScript is highly performant, it is everywhere, will be immediately useful and supports most programming paradigms, e. g. functional programming (you’re not stuck with imperative and object oriented programming like in Phyton).

My personal choice

On my shortlist I had Racket, Lua, Haskell and Common Lisp. Ultimately, I started with Common Lisp. So what’s my motivation for that?

I’ve looked into many books, and it turned out that I have a favorite textbook on programming and I want to work with that one first. It is a great introduction and has many exercises. Secondly, I favour convenience and exploration. The interactive Common Lisp development workflow is far more advanced than anything I’ve seen so far.

Lisp goes very well with Emacs. You don’t have to use Emacs, there are plugins for Atom, VSCode, Vim and others. Emacs is written in Emacs Lisp, another programming language within the Lisp family. Emacs Lisp and Common Lisp are similar enough to avoid confusion while switching back and forth beetween both. Compare the syntax of the four most used Lisps. Actually, I’m writing more Emacs Lisp than Common Lisp at the moment. But that’s a story for another time.
